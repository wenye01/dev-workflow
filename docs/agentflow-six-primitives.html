agentflow · primitive basis

六原语如何协作完成一个任务

把多 Agent 协作拆到「再往下拆只会增加系统复杂度」为止，得到六个最小职责单位。它们各自的内容、彼此的五个交互函数、以及一条完整任务在这套范式里如何跑完——全部可交互。

原语判据：原语 = 多 Agent 协作下难以继续拆分的职责单位。往下拆反而增加复杂度的，才保留为原语；能由其它原语合成的是视图（如 Session = fold(events, artifacts)，故不列为原语）；Actor 不运行不产出，只是 Activation 的 target 参数，也降级。

一、六原语各自的内容

每个原语有唯一职责，下面列出它「内部承载什么」和「刻意不承担什么」。

PRIMITIVE 01

Activation

执行轴 · 一次受控委派
把目标交给谁、在什么约束下、期望什么产物——协作的原子动作。
内部内容
  • id唯一标识
  • target指向谁（原 Actor 退化进来）
  • objective目标
  • input_artifacts输入引用
  • expected_outputs输出契约
  • capability_grants→ Capability
  • lifecycletimeout / cancel / retry
  • budget_class预算类别
  • metadatarecipe / phase / group
不做：不管 Agent 内部每一条 bash / 工具调用（那是内部 trace）。
PRIMITIVE 02

Capability

约束轴 · 静态能力边界
这次 Activation 被允许用什么工具、资源、预算。prompt 不是边界，它才是。
内部内容
  • file_read_scope可读路径
  • file_write_scope可写路径
  • toolsbash / git / mcp…
  • networkbool
  • sandboxnone / worktree / container
  • budget单次 token / wall-time 上限
  • approval_required是否需人工
不做：不决定业务目标（那是 Recipe），不动态升级（那是 Policy）。
PRIMITIVE 03

Artifact

状态轴 · 跨边界稳定产物
唯一能被后续 Activation 引用的稳定内容。不外化就不算系统状态。
内部内容
  • ref稳定路径
  • schema_id结构契约
  • content_hash内容寻址
  • producer哪个 Activation 产出
  • input_refs依赖了哪些
  • payload结构化内容
  • viewsmarkdown / summary 渲染
不做：不承载隐式上下文，不保存过程（过程归 Event）。
PRIMITIVE 04

Event

事实轴 · append-only 事实流
只追加、不改写。是恢复、审计、重放、成本核算的唯一来源。
内部内容
  • typeactivation.started 等
  • timestamp顺序号
  • activation_ref归属
  • payload事实细节
不做：不作为可变状态表；与 Artifact 正交（事实流 vs 状态快照）。
PRIMITIVE 05

Recipe

编排轴 · 读状态 → 提议下一步
纯函数。读取 fold 出的 state，提议下一个/组 Activation，直到 Done。
内部内容
  • read(state)只读 fold 结果
  • → Directivepropose / done / stop
  • kinddeterministic / recorded-dynamic
不做：不直接调模型做控制流（LLM 决策先固化为 Event 再被读取），不裁决（那是 Policy）。
PRIMITIVE 06

Policy

裁决轴 · 准许 / 升级 / 停止
动态裁决。根据风险/失败/分歧信号，决定 Activation 放行、升级复杂度还是停止。
内部内容
  • capability_gate权限检查
  • budget_gate全局预算
  • risk_gate风险检查
  • escalation_rules低置信/多轮失败→升级
  • stop_conditions终止条件
不做：不提议做什么（那是 Recipe）；静态权限归 Capability，全局预算归这里。

二、原语间的五个交互函数

整套范式只需要这五个函数。前三个由 Activation 触发，后两个是内核循环的每一拍。

1
activate(target, objective, capability) → Activation
委派：把目标交给某执行者，附上能力边界
2
write(activation, artifact) → Artifact
产出：外化为可被后续引用的稳定状态
3
emit(activation, event) → Event
留痕：append-only 追加事实
4
decide(recipe, fold(events, artifacts)) → [Activation] | Done | Stop
编排：纯函数读状态，提议下一步
5
gate(policy, activation, signals) → admit | escalate | stop
裁决：是否放行 / 升级 / 终止
关键不变量： decide() 与 gate() 只读 fold(events, artifacts)，永不直接调模型。 所以即使整个 workflow 是「AI 驱动」的，编排与裁决层依然 100% 可重放——LLM 的任何决策都必须先固化成 Event/Artifact，再被这两个纯函数读取。
